项目代码编写流程

准备阶段

  • 根据 项目目录结构 创建对应文件夹与文件，可以编写通用模板，以便可以直接复制从而创建新项目
  • 在 doc 文件夹放置算法的设计思路文件与 MATLAB 预演代码（假设已经在 MATLAB 进行过实验）
  • 确定对外接口头文件 ProjectName.h 和对应源文件 ProjectName.c（源文件此时为一系列空函数）
  • 预先准备好测试数据与预期的结果数据，放入 data 文件夹（一般情况下应与 MATLAB 预演版本数据保持一致）
  • 修改 example.cpp 文件，调用 ProjectName.c 的空文件，完成整个项目的框架
  • 修改 CMakeLists.txt 文件（修改版本号，如果调用了外部库则需要添加依赖库）
  • 调试 example.cpp 代码 ，确保在框架的正常运行

编码阶段

  • 在 src 文件夹下编写功能代码
  • 在 test/unit_test 文件夹下编写单元测试代码
  • 调试 src 和 test/unit 文件夹下的代码直到测试通过
  • 在 ProjectName.c 中调用 src 各个模块的子函数
  • 调试 example.cpp 代码，直至输出与预期相符
  • 编写 Readme.md 例子程序说明文档，介绍本项目的功能、参数、注意事项、函数调用流程等

测试阶段

  • 在 test/integration_test 中编写集成测试，主要测试接口输入参数的异常处理，函数调用顺序的异常处理
  • 在 test/system_test 中编写系统测试，主要测试异常数据导致的错误

C语言编码规范

主要参考 PEP7，并做了适当调整

排版

  1. 使用 4 个空格的缩进，完全没有制表符。
  2. 任何行都不应超过 79 个字符。
  3. 任何行都不应该以空格结尾
  4. 函数定义风格：‘{’ 不需独占一行，‘}’ 需独占一行，同时与引用代码块的语句左对齐，局部变量声明后空行。
  5. 代码结构：关键字（如if、for）和后面的左括号之间有一个空格；括号内没有空格；即使 C 允许省略，任何地方都需要大括号，但不要将它们添加到您没有修改的代码中。所有新的 C 代码都需要大括号。大括号的格式应如下所示：
  6. return 语句不应该有多余的_括号_
  7. 函数和宏调用风格：foo(a, b, c) - 左括号前无空格，括号内无空格，逗号前无空格，每个逗号后一个空格。
  8. 始终在赋值、布尔和比较运算符周围放置空格。在使用大量运算符的表达式中，在最外层（最低优先级）运算符周围添加空格。

  11. 在函数、结构定义和函数内部的主要部分周围放置空白行。
  12. 注释位于他们描述的代码之前。

命名

单词命名

  1. 不允许使用单个字符（如 i j k）命名，作为局部循环变量除外
  2. 命名单词为全英文，不允许使用中文拼音
  3. 函数/方法名：一般采用动宾结构命名。如果是类中的方法名，可以直接使用动词结构
  4. 避免使用无意义或含义不清的动词为函数命名。如 process、handle
  5. 外部可见的标识符应加上模块名，防止编译、链接时产生冲突
  6. 用正确的反义词组命名具有互斥意义的变量或相反动作的函数
word 1 word 2 word 1 word 2 word 1 word 2
add remove begin end create destroy
insert previous first last get release
next decrement put get add delete
lock unlock open close min max
old new start stop increment decrement
cut paste up down

命名风格

  1. 包/文件夹：采用小驼峰命名规则，比如 ecgBinaryClassification
  2. 文件名：下划线方式，比如 ecg_binary_classification
  3. 类/结构体/自定义类型名：大驼峰法，比如 EcgBinaryClassification
  4. 函数/方法名：采用小驼峰或下划线，这里建议采用小驼峰方式
  5. 局部变量名：采用小驼峰或下划线，这里建议采用小驼峰方式
  6. 静态变量名：采用在局部变量规则前加 s_ 的形式
  7. 全局变量名：采用在局部变量规则前加 g_ 的形式
  8. 内部符号：采用在符号前加 _ 的形式
  9. 私有符号：采用在符号前加 __ 的形式
  10. 常量：全部大写，单词与单词之间采用下划线连接

注释

源程序有效注释量必须在20％以上

文件注释

函数注释

  1. 函数，原则上都需要加注释，尤其是接口函数和逻辑比较复杂的函数
  2. 函数的返回值注释要清楚、明了

变量注释

  1. 变量：要求静态变量和全局变量必须加注释
  2. 命名中若使用特殊约定或缩写，必须有注释说明

其他注释

  1. 语句：有技巧性的代码语句，必须添加注释
  2. TODO：尽量不使用 “待完善” “待明确”等模糊词，而使用相对明确的方式，如
    # TODO(author) 添加低通滤波功能.

函数

  1. 每个函数完成一个功能
  2. 函数的入参和出参不允许直接传递结构体对象，需改为指针
  3. 行数限制 200 行，如果某个函数有效代码超过 200 行，考虑分拆。
  4. 语句嵌套：主要是 if...else ,, while ， for 等，要求嵌套不超过 3 层， switch...case 不超过五层。
  5. 明确函数功能，精确（而不是近似）地实现函数设计
  6. 对所调用函数的错误返回码要仔细、全面地处理
  7. 编写可重入函数时，若使用全局变量，则应通过关中断、信号量（即P、V 操作）等手段对其加以保护
  8. 检查函数所有参数输入的有效性
  9. 检查函数所有非参数输入的有效性，如数据文件、公共变量等
  10. 防止把没有关联的语句放到一个函数中。在编程时，经常遇到在不同函数中使用相同的代码，需要把这些代码提出来，并构成一个新函数。
  11. 减少函数本身或函数间的递归调用
  12. 不能随意改变与其它模块的接口
  13. 数入参为指针或者有其他有限制的值时必须加入入参检测

语句

循环

  1. 循环体内工作量最小化
  2. 避免循环体内含判断语句，应将循环语句置于判断语句的代码块之中
  3. 在多重循环中，应将最忙的循环放在最内层
  4. 要防止差1 错误。此类错误一般是由于把 <=** 误写成 **<** 或 **>= 误写成 > 等造成的，由此引起的后果，很多情况下是很严重的
  5. for 循环中不要使用 return 来退出双重循环，可以使用标志位判断来退出双重循环

临界资源操作

临界资源包括内存、文件、句柄等

  1. 过程/函数中分配的内存，在过程/函数退出之前要释放
  2. 过程/函数中申请的（为打开文件而使用的）文件句柄，在过程/函数退出之前要关闭
  3. 申请了临界资源后只有一个出口 return 来退出过程/函数，以防止忘记关闭临界资源
  4. 防止内存/数组操作越界

其他

  1. 用括号明确表达式的操作顺序，避免使用默认优先级
  2. 判断条件的执行语句必须用花括号括起来
  3. 不允许直接使用数字，需用宏定义或变量替换
  4. 关系较为紧密的代码应尽可能相邻
  5. 要使用难懂的技巧性很高的语句，除非很有必要时。且必须加注释
  6. 尽量减少没有必要的数据类型默认转换与强制转换
  7. 初始化有关变量及运行环境，防止未经初始化的变量被引用
  8. 判断性语句，把常量放在表达式的左边
  9. if 语句尽量加上 else 分支，对没有 else 分支的语句要小心对待；switch 语句必须有 default 分支
  10. 使用变量时要注意其边界值的情况。如 C 语言中字符型变量，有效值范围为 -128 到 127
  11. 求余和除法要做除数的0值判断
  12. 用 free() 函数时需要对入参判断其是否为空指针

自定义类型

  1. 结构的功能要单一
  2. 不要设计面面俱到、非常灵活的数据结构
  3. 仔细设计结构中元素的布局与排列顺序，使结构容易理解、节省占用空间，并减少引起误用现象
  4. 避免使用 BOOL 类型

变量

  1. 申请的变量必须初始化
  2. 严禁使用未经初始化的变量作为右值
  3. 使用静态变量和全局变量时，仔细定义并明确公共变量的含义、作用、取值范围及公共变量间的关系

代码测试

  1. 单元测试要求至少达到语句覆盖。
  2. 单元测试开始要跟踪每一条语句，并观察数据流及变量的变化。
  3. 清理、整理或优化后的代码要经过审查及测试。
  4. 代码版本升级要经过严格测试。
  5. 坚持在编码阶段就对代码进行彻底的单元测试，不要等以后的测试工作来发现问题
  6. 仔细设计并分析测试用例，使测试用例覆盖尽可能多的情况，以提高测试用例的效率。
  7. 尽可能模拟出程序的各种出错情况，对出错处理代码进行充分的测试。
  8. 仔细测试代码处理数据、变量的边界情况
  9. 不应通过“试”来解决问题，应寻找问题的根本原因。
  10. 坚持在编码阶段就对代码进行彻底的单元测试，不要等以后的测试工作来发现问题

其他

  1. 去除编译中的所有警告
